About this
manual

Welcome to Version 2.0 of the iMRV installation guide. This release covers the modern Frappe v16 stack with the unified install.sh bootstrap, the Vue 3 single-page front-end, and the Forest-and-Sage editorial system that runs across the desk and SPA.

Whether you're a professional installer or a first-time user, this manual covers everything you need to install and configure the SI iMRV Tool reliably — from a fresh laptop to a hardened production server.

Inside, you'll find system requirements, two install paths (one-command and manual), daily lifecycle scripts, the directory layout, production hardening (domain + TLS), and a brief Railway / container deployment overview. We recommend reading the manual end-to-end before beginning.

§ 1  ·  Introduction

Section 1

Introduction

The Solomon Islands' Integrated Monitoring, Reporting and Verification (iMRV) Tool is built on the Frappe Framework — a full-stack, batteries-included Python web framework with a built-in ORM over MariaDB.

1.1 What ships in this repository

This repository (iMRV-Solomon-Islands) contains three layers that install side-by-side and serve a single coherent product:

LayerDescription
mrvtools The main Frappe app — projects, adaptation, mitigation, GHG inventory, climate finance, and tracking reports. Ships two Frappe modules: Mrvtools and GHG Inventory.
frappe_side_menu A second Frappe app that renders the desk's floating navigation drawer (top-left burger button → click-triggered slide-in panel over a translucent forest backdrop). Installed as a separate bench app.
frontend (SPA) Vue 3 + Vite + Tailwind + Frappe UI single-page app. Built into mrvtools/public/frontend/ and served by Frappe at /frontend/* (Home, About, Projects, Climate Change Division, Reports, Support, Knowledge Resource, What's New, Login).

1.2 Frappe Framework & Bench

Frappe Framework uses the following services for running the SI iMRV Tool:

#ServicePurpose
IMariaDBRelational database (10.6 or newer)
IIFrappe (Python)Web framework + ORM + business logic
IIIRedisCache, queue, and socket-io broker (3 instances)
IVNode.jsRealtime socket-io server + asset build
VNginx (prod)Reverse proxy + static file server

Frappe is Python-based and uses an isolated virtualenv at frappe-bench/env/. Deployment is handled by Bench, a CLI utility that installs, manages, and runs all the services required by Frappe. Bench is multi-tenant — each tenant has its own database and files, served from a separate folder under sites/.

§ 1.3  ·  Architecture

Section 1.3

Architecture

The diagram below mirrors the on-disk frappe-bench directory layout and the runtime stack. Browser clients reach Frappe through Nginx in production (or directly through Werkzeug in development).

Browser SPA at /frontend/*  ·  Desk at /app/*  ·  Login & auth
Nginx (prod)   ·   Vite dev server (dev) /api · /method · /assets · /files · /private · /app · /frontend · /socket.io
frappeframework
mrvtoolsmain app
frappe_side_menudrawer
frontendVue 3 SPA
Python venv3.10+
Nodesocket.io
Redis ×3cache · queue · socket
Workersbackground jobs
MariaDB   ·   Sites volume (public/files, private/files, backups) One database per site  ·  multi-tenant by design

Routing handoff

The Frappe webserver maps /frontend/<path> to the SPA shell template, then createWebHistory('/frontend') in the Vue router takes over. /app/* is the Frappe desk (logged-in users); /frontend/* is the public-facing SPA.

Build artefact convergence

The SPA build writes into mrvtools/public/frontend/, and Frappe serves anything under mrvtools/public/ at /assets/mrvtools/.... The desk-injected CSS/JS and the SPA bundle therefore coexist under the same public tree.

§ 3  ·  Quick Start

Section 3  ·  Recommended path

Quick start —
one command

The repository ships an idempotent install.sh that takes a fresh macOS, Ubuntu, or Windows-via-WSL2 laptop to a working iMRV environment in a single command. It installs every OS package, initialises bench, creates the site, installs both apps, builds the SPA, and (in dev) restores the demo database.

1

Clone the repository

Pull the monorepo (which contains both mrvtools and frappe_side_menu).

$ git clone https://github.com/gotaker/iMRV-Solomon-Islands.git
$ cd iMRV-Solomon-Islands
2

Pick a mode

Pass --dev for a local development environment (Vite dev server, demo DB restored, ignore_csrf flipped on) or --prod for a supervisor/nginx production stack on Ubuntu.

# Dev (recommended for first install)
$ MYSQL_ROOT_PASSWORD=<your-pw> ./install.sh --dev

# Prod (Ubuntu only — provisions supervisor + nginx)
$ sudo MYSQL_ROOT_PASSWORD=<your-pw> ./install.sh --prod
3

Wait, then sign in

The first run takes 10–20 minutes (most of it is bench init compiling Python wheels and pulling the Frappe v16 source). When it finishes, the script prints the URL and credentials.

Default site: http://mrv.localhost:8000  ·  user Administrator  ·  password from ADMIN_PASSWORD (defaults to admin).

options

Modes & flags

VariableDefaultPurpose

⚠ Idempotence

install.sh is safe to re-run — every step checks for prior completion and skips. It will not overwrite an existing site or drop your database. To start from scratch, remove ~/frappe-bench/sites/mrv.localhost/ and the corresponding MariaDB database manually first.

§ 4  ·  Manual install

Section 4  ·  Alternate path

Manual
installation

Use this path if you want to understand exactly what install.sh does, or if you're integrating with an existing bench. All commands below run from your bench root (typically ~/frappe-bench) unless noted.

I

Install Frappe Bench

Follow the upstream Frappe installation guide for your OS: frappeframework.com/docs/v16/user/en/installation. Pin the branch to version-16.

$ bench init frappe-bench --frappe-branch version-16 --python python3.14
$ cd frappe-bench
II

Create the site

This creates a new database — you'll be prompted for the MariaDB root password and asked to set the Administrator password. Pick something memorable; you'll need it on first sign-in.

$ bench new-site mrv.localhost
III

Add the iMRV apps

Both apps live in the same monorepo. Clone it, then point bench get-app at the local paths.

$ cd ~
$ git clone https://github.com/gotaker/iMRV-Solomon-Islands.git
$ cd ~/frappe-bench

# Main app
$ bench get-app mrvtools ~/iMRV-Solomon-Islands

# Drawer / side menu
$ bench get-app frappe_side_menu ~/iMRV-Solomon-Islands/frappe_side_menu
IV

Install both apps onto the site

Order matters: install mrvtools first so its master data and single-doctypes are seeded, then frappe_side_menu on top.

$ bench --site mrv.localhost install-app mrvtools
$ bench --site mrv.localhost install-app frappe_side_menu
$ bench --site mrv.localhost migrate

§ 4  ·  Manual install

Section 4  ·  continued

Build
the SPA

V

Install front-end dependencies

The frontend/ directory holds the Vue 3 SPA. It builds into mrvtools/public/frontend/ and is served by Frappe at /frontend/*.

$ cd ~/iMRV-Solomon-Islands/frontend
$ yarn install
VI

Build the production bundle

Builds the SPA, writes assets into mrvtools/public/frontend/, and copies index.html to mrvtools/www/frontend.html — which is what Frappe serves under /frontend.

$ yarn build

To verify the build locally without a bench, serve the actual output directory:

$ python3 -m http.server -d ~/iMRV-Solomon-Islands/mrvtools/public/frontend 8090

Then open http://localhost:8090/. Don't use vite preview — it serves from frontend/dist with the dev base URL and won't reflect what Frappe ships.

VII

Start the bench

From your frappe-bench root:

$ cd ~/frappe-bench
$ bench start

The web server listens on :8000; Vite (when running) listens on :8080; the socket-io server uses :9000. The dev SPA proxies /api, /method, /assets, /files, /private, /app, /login, /logout, and /socket.io to the bench at :8000.

SPA dev workflow

For active SPA development, run bench start on :8000 and yarn dev in frontend/ on :8080 in parallel. The Vite plugin reads your bench's common_site_config.json automatically so the proxy targets the right host — visiting http://mrv.localhost:8080 proxies to http://mrv.localhost:8000.

§ 5  ·  Daily lifecycle

Section 5

Daily
lifecycle

Once the stack is installed, day-to-day stop / start is driven by two scripts at the repo root: start.sh and shutdown.sh. Both are idempotent and honour BENCH_DIR, SITE_NAME, and DRY_RUN=1.

Starting the stack

CommandWhat it does
./start.sh --dev Brings up MariaDB & Redis (if not running), starts bench start in the background, and launches Vite on :8080. Best for active SPA work.
./start.sh --prod Brings up MariaDB & Redis only — supervisor + nginx are managed by systemd in production. No Vite.

Stopping the stack

CommandWhat it does
./shutdown.sh Stops bench start and Vite. Sweeps any orphan listeners on the bench's actual ports (read from common_site_config.json). Leaves MariaDB/Redis up.
./shutdown.sh --full Same as above plus stops MariaDB and Redis. Use before laptop reboot or to free the ports completely.

Front-end commands

CommandWhat it does
yarn dev Vite dev server on :8080 with HMR. Run from frontend/.
yarn build Production bundle into mrvtools/public/frontend/.
yarn lint Prettier check + ruff (matches CI).

⚠ After pulling new code

Always run bench --site mrv.localhost migrate after a git pull that includes new doctype JSON or migration patches. The harness at tests/run.sh will fail loudly if you skip this — that's the v16 schema gate firing.

§ 6  ·  Directory structure

Section 6

Directory
structure

When you initialise a new frappe-bench directory, you'll have a structure similar to the tree below. The four top-level folders below correspond to the four sub-sections of this manual.

frappe-bench/ ├── apps/ # All Frappe-based apps live here │ ├── frappe/ # Framework │ ├── mrvtools/ # Main iMRV app │ └── frappe_side_menu/ # Drawer app ├── config/ # Redis & supervisor config │ ├── redis_cache.conf │ ├── redis_queue.conf │ └── redis_socketio.conf ├── env/ # Python virtualenv ├── logs/ # Per-process log files ├── Procfile # Read by `bench start` └── sites/ # Tenant data (one folder per site) ├── apps.txt # Apps known to bench ├── assets/ # Built JS/CSS for desk ├── common_site_config.json └── mrv.localhost/ # A site (the database is named after this) ├── site_config.json ├── private/files/ └── public/files/

6.1 apps

Frappe and every Frappe-based app live here. bench new-app <name> bootstraps a new app in this directory. Custom apps are edited in place.

6.2 sites

Each subdirectory is a tenant — its own MariaDB database and its own files. Created by bench new-site <name>; named after the site's HTTP host header.

6.3 config

Three Redis instances run side-by-side: one for cache, one for the job queue, one for socket-io. Their configs live here. Don't edit by hand — bench setup redis regenerates them from templates.

6.4 env

The Python virtualenv. Frappe-based apps and Python dependencies install here. Activate with source env/bin/activate if you need to run pip or python in the bench's exact environment.

For the upstream reference, see frappeframework.com/docs/v16/user/en/basics/directory-structure.

§ 7  ·  Production hardening

Section 7

Production —
domain & TLS

Once ./install.sh --prod (or the manual bench setup production) has provisioned supervisor + nginx, you'll add your domain and provision a TLS certificate. The flow below works on Ubuntu 22.04 LTS.

1

Enable production mode

$ bench config dns_multitenant on
$ sudo bench setup production <system_user>
$ bench setup nginx
$ sudo service nginx restart
2

Add your domain

Map a domain to the site. Bench will prompt for which site to attach.

$ bench setup add-domain "imrv.example.gov.sb"
3

Install Certbot & provision TLS

The recipe below uses snap on Ubuntu — the upstream-supported path for keeping certbot current. If snap isn't present: sudo apt-get install snapd.

$ sudo snap install core
$ sudo snap refresh core
$ sudo snap install --classic certbot
$ sudo ln -s /snap/bin/certbot /usr/bin/certbot

# Provision the cert against your domain
$ sudo certbot --nginx

# Reload nginx
$ sudo service nginx restart
4

Or do all of the above with one flag

install.sh can drive the whole flow when given PROD_DOMAIN and PROD_ENABLE_TLS=1:

$ sudo MYSQL_ROOT_PASSWORD=<pw> \
    PROD_DOMAIN=imrv.example.gov.sb \
    PROD_ENABLE_TLS=1 \
    ./install.sh --prod

⚠ DNS first

Certbot's HTTP-01 challenge needs your A/AAAA records pointing at the server before you provision the cert. If DNS hasn't propagated, certbot will fail with "challenge did not complete". Wait, then re-run.

§ 8  ·  Container deployment

Section 8

Container /
Railway

A staging / demo environment ships as a single container, built from Dockerfile at the repository root and launched on Railway. Locally, you can run the same image via Docker Compose to verify changes before deploy.

Build & run locally

# From the repo root
$ docker compose -f deploy/railway/docker-compose.local.yml up --build

The image bundles: Frappe v16 + mrvtools + frappe_side_menu + the SPA bundle + wkhtmltopdf + nginx + supervisor. entrypoint.sh seeds the site on first boot, applies migrations on every boot, and starts gunicorn behind nginx.

Deploy to Railway

The full operational runbook lives at deploy/railway/README.md. In short:

  • Push to your Railway-connected git remote — Railway builds the Dockerfile.
  • Configure environment: SITE_NAME, ADMIN_PASSWORD (first boot only), SAMPLE_DB_URL (a GitHub Release asset).
  • Mount a Railway volume at /home/frappe/frappe-bench/sites for site data.
  • Verify with the bench framework: ./bench/bench.sh --target=railway-preview --tier=a.

Five non-obvious invariants

1. /files/ must be an Nginx alias, not a proxy_pass to gunicorn — Frappe's WSGI returns 404 for public file URLs in this single-tenant setup.
2. The sites volume is seeded from a baked-in template only when empty. A volume remount that loses files won't re-trigger seeding.
3. ADMIN_PASSWORD is read only on the very first boot. Rotate via bench set-admin-password inside the container thereafter.
4. The Nginx /assets/ block uses Cache-Control: no-cache, must-revalidate + ETag — intentional, so hot-patched edits via docker cp are picked up on the next request.
5. frappe_side_menu lives at a nested path inside the container — the import frappe_side_menu.frappe_side_menu.api resolves to apps/frappe_side_menu/frappe_side_menu/frappe_side_menu/api.py (note the doubled directory). Patch the inner copy, not the sibling.

§ 9  ·  Verification

Section 9

Verifying
the install

Two parallel test loops live in this repo. Run both before declaring an install complete — they catch different classes of failure.

9.1 Pytest harness — code-level integration

The primary local test loop. Five layers (data, integration, ui, regression, security) covering ~50 tests against a fresh bench on port 8001 — decoupled from your dev bench on :8000.

$ ./tests/run.sh                     # all layers, ~2–3 min
$ ./tests/run.sh --layer integration  # single layer
$ ./tests/run.sh --fast               # layers 1+2+5 (no UI), ~45s
$ ./tests/run.sh --update-golden      # regenerate Layer 4 snapshots

9.2 Bench framework — deploy verification

Two-tier journey-based testing against deployed targets (local Docker, Railway preview). A-tier (deterministic Playwright YAMLs) is a hard gate; B-tier (LLM-judged rubric scoring) is observability.

$ ./bench/bench.sh --target=local-docker --tier=a --dry-run
$ ./bench/bench.sh --target=local-docker --tier=a

9.3 Smoke test by hand

  • Visit http://mrv.localhost:8000/frontend/home — the SPA homepage should load with the Forest-and-Sage palette and the Solomon Islands flag in the navbar.
  • Click the burger button (top-left on the desk) — the floating drawer should slide in over a translucent forest backdrop.
  • Sign in as Administrator — landing page should be /app/main-dashboard.
  • Navigate to a Tracking Report and click Download PDF — a Forest-and-Sage editorial PDF should render.
